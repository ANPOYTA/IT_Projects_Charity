## 1. Загальна інформація

- \*\*Тема проєкту:\*\* Веб-платформа для благодійних/волонтерських зборів[cite: 6].
- \*\*Мета:\*\* Вирішення проблем довіри до зборів через прозору верифікацію та централізацію звітності[cite: 8].

## 3. Матриця розподілу відповідальності (RACI)

Нижче наведено розподіл обов'язків згідно з моделлю RACI, де:

- **R (Responsible)** — Виконавець.
- **A (Accountable)** — Відповідальний.
- **C (Consulted)** — Консультує.
- \*\*I (Informed)\*\* — Інформується[cite: 26, 31].

| Основні завдання проєкту                      | PM/FE Dev (Анна П.) | BA/BE Dev (Маргарита П.) | RM/QA (Анна М.) |
| :-------------------------------------------- | :-----------------: | :----------------------: | :-------------: |
| Підготовка Project Charter та діаграми Ганта  |      **R, A**       |            C             |                 |
| Створення Scope Statement та WBS              |          A          |          **R**           |        C        |
| Підготовка Risk Management Plan та Risk Log   |          A          |            I             |      **R**      |

| Створення клієнтського інтерфейсу (FE)        |      **R, A**       |            C             |        I        |

| Фінальний реліз платформи                     |          A          |          **R**           |      **R**      |

> \*Примітка: Дані базуються на таблиці завдань проєкту[cite: 23].\*

## 4. Обґрунтування розподілу

1.  \*\*Чітка відповідальність:\*\* За кожне завдання відповідає лише одна людина (Accountable), що унеможливлює розмиття відповідальності[cite: 28, 30].
2.  \*\*Ефективна комунікація:\*\* Матриця визначає, хто надає консультації (C), а хто лише отримує статус (I), що економить час на мітингах[cite: 31, 33].
3.  \*\*Незалежний контроль якості:\*\* Роль QA (Анна Міллер) відокремлена від написання коду, що забезпечує об'єктивність тестування[cite: 35].
4.  \*\*Паралельна розробка:\*\* Чітке розмежування R (виконавців) дозволяє фронтенду та бекенду працювати без взаємних блокувань[cite: 36, 37].
