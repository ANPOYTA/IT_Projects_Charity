# RACI-модель проєкту: Веб-платформа для волонтерських зборів

## 1. Загальна інформація

- **Тема проєкту:** Веб-платформа для благодійних/волонтерських зборів.
- **Мета:** Вирішення проблем довіри до зборів через прозору верифікацію та централізацію звітності.

## 2. Команда та ролі

| Учасник                | Ролі (Управлінська / Технічна) | Основні обов'язки                                           |
| :--------------------- | :----------------------------- | :---------------------------------------------------------- |
| **Анна Пойта**         | PM / Frontend Developer        | Таймінг, Хартія проєкту, верстка сторінок, UI/UX.           |
| **Маргарита Панченко** | BA / Backend Developer         | Збір вимог (WBS), серверну частина, бази даних, API банків. |
| **Анна Міллер**        | Risk Manager / QA & Security   | План ризиків, тестування безпеки платежів, захист від атак. |

## 3. Матриця розподілу відповідальності (RACI)

Нижче наведено розподіл обов'язків згідно з моделлю RACI, де:

- **R (Responsible)** — Виконавець.
- **A (Accountable)** — Відповідальний.
- **C (Consulted)** — Консультує.
- **I (Informed)** — Інформується.

| Основні завдання проєкту                      | PM/FE Dev (Анна П.) | BA/BE Dev (Маргарита П.) | RM/QA (Анна М.) |
| :-------------------------------------------- | :-----------------: | :----------------------: | :-------------: |
| Підготовка Project Charter та діаграми Ганта  |      **R, A**       |            C             |                 |
| Створення Scope Statement та WBS              |          A          |          **R**           |        C        |
| Підготовка Risk Management Plan та Risk Log   |          A          |            I             |      **R**      |
| Розробка архітектури та серверної логіки (BE) |                     |         **R, A**         |        C        |
| Створення клієнтського інтерфейсу (FE)        |      **R, A**       |            C             |        I        |
| Інтеграція платіжних систем (банки/шлюзи)     |          C          |          **R**           |        C        |
| Тестування безпеки платежів та пошук багів    |          I          |            I             |    **R, A**     |
| Фінальний реліз платформи                     |          A          |          **R**           |      **R**      |

> _Примітка: Дані базуються на таблиці завдань проєкту._

## 4. Обґрунтування розподілу

1.  **Чітка відповідальність:** За кожне завдання відповідає лише одна людина (Accountable), що унеможливлює розмиття відповідальності.
2.  **Ефективна комунікація:** Матриця визначає, хто надає консультації (C), а хто лише отримує статус (I), що економить час на мітингах.
3.  **Незалежний контроль якості:** Роль QA (Анна Міллер) відокремлена від написання коду, що забезпечує об'єктивність тестування.
4.  **Паралельна розробка:** Чітке розмежування R (виконавців) дозволяє фронтенду та бекенду працювати без взаємних блокувань.
